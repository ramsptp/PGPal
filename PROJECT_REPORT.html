1. Abstract

PGPal is a web-based Paying Guest (PG) Management System developed using the MEAN stack — MongoDB, Express.js, AngularJS (v1.x), and Node.js. The system provides a comprehensive digital platform for managing PG accommodations, covering room allotment, tenant registration, rent tracking, maintenance complaint management, and notice board communication. A distinguishing feature is the Roommate Matcher, a logic-driven algorithm that compares tenant lifestyle preferences using a weighted scoring model to suggest compatible roommates. The application implements JWT-based authentication with role-based access control to separate administrative and tenant functionality.

2. Introduction

Paying Guest (PG) accommodations are a widely prevalent form of housing in India, particularly among students and working professionals. Managing a PG facility manually — tracking room occupancy, collecting rent, handling maintenance complaints, and communicating notices — is time-consuming and error-prone.

PGPal digitises this entire workflow into a single web application. Administrators gain a real-time overview of their facility through a data-rich dashboard, while tenants can manage their stay, track rent payments, and communicate maintenance issues from any web browser.

The project is built strictly on the MEAN stack as required, using AngularJS 1.x (with ng-app, ng-controller, $scope, and $http) for the frontend, Node.js with Express.js for the backend API, and MongoDB Atlas as the cloud-hosted NoSQL database. All data exchange between the frontend and backend is performed in JSON format via RESTful API endpoints.

3. Problem Statement

Traditional PG management relies on physical registers, manual receipts, and informal communication. This leads to several challenges:

  • Difficulty tracking room occupancy and vacancy status in real time.
  • Lack of a formal channel for tenants to report maintenance issues.
  • No structured system for rent payment acknowledgment.
  • No mechanism for matching compatible roommates, leading to friction among co-tenants.
  • Communication between admin and tenants is informal and unreliable.

PGPal addresses all of these problems through a structured, role-based web application.

4. Objectives

  • Develop a full-stack web application using the MEAN stack for PG facility management.
  • Implement role-based access control distinguishing Admin and Tenant users.
  • Provide Admin with CRUD operations for rooms, tenant management, maintenance oversight, and notice posting.
  • Enable Tenants to view their room details, submit maintenance requests, check notices, submit rent receipts, and edit their preferences.
  • Implement a Roommate Matcher algorithm using weighted comparison of JSON preference attributes.
  • Use MongoDB Atlas as the cloud database with Mongoose ODM for schema enforcement.
  • Secure all API routes with JSON Web Tokens (JWT).

6. System Architecture

PGPal follows a three-tier client-server architecture:

  • Presentation Layer (Frontend): AngularJS 1.x SPA served as static files. Client-side routing via ngRoute. All API communication through AngularJS $http service.
  • Application Layer (Backend): Express.js server exposes RESTful API endpoints. JWT middleware validates every protected route. Business logic (e.g., Roommate Matcher scoring) resides here.
  • Data Layer (Database): MongoDB Atlas cluster stores all documents. Mongoose provides schema validation and lifecycle hooks.

8. Module Description

8.1 Authentication Module

Both admins and tenants are stored in the same Tenant collection, differentiated by a role field. On login, the server validates credentials using bcrypt.compare() and issues a signed JWT with a 24-hour expiry containing the user's ID and role. The token is stored in localStorage on the client.

An AngularJS HTTP interceptor automatically attaches the token as a Bearer header to every outgoing $http request. A response interceptor detects HTTP 401 responses (expired/missing token) and automatically redirects the user to the login page, clearing local storage.

8.2 Admin Module

  • Dashboard: Displays total rooms, occupied/vacant count, bed-based occupancy rate (%), expected monthly revenue (pricePerMonth × occupiedBeds), confirmed rent collected this month, total tenants, unassigned tenants, pending maintenance complaints, and pending rent receipts. Recent maintenance requests and recently joined tenants are shown in side-by-side tables.
  • Room Management: Full CRUD for rooms. Room type automatically determines bed capacity via Mongoose hook. Admin can search/filter rooms and click a 👥 button to view current occupants of any room.
  • Tenant Management: View all tenants with their room preferences shown as tags. Admin can assign/reassign rooms, vacate tenants, edit tenant details (name, phone, rent due day), and print the tenant list. Inline row editing replaces the selected row with editable fields.
  • Maintenance Log: View all complaints across all tenants. Admin can update status (Pending → In Progress → Resolved) and add a note that tenants will see.
  • Notice Board: Post general text notices and delete old ones.
  • Rent Receipts: Review all payment claims. Confirm or reject each with an optional note to the tenant.

8.3 Tenant Module

  • Dashboard: Shows tenant's name and contact, assigned room details and price, days until next rent due date (computed dynamically), open complaint count, last receipt status, latest 3 notices, and recent maintenance requests. Clicking the profile card opens an inline form to edit name, phone, and password. A separate section allows editing roommate preferences and room type preferences.
  • Maintenance Requests: Submit new complaints with title and description. View all own requests with current status and admin notes.
  • Notice Board: Read-only view of all general notices posted by admin.
  • Rent Payments: Submit payment claims (month, amount, UPI/bank reference). System prevents duplicate submissions for the same period. View full payment history with admin decision status.
  • Roommate Matcher: Displays compatibility scores with all other tenants sorted from most to least compatible.

9. Roommate Matcher Algorithm

The Roommate Matcher is the unique special-logic feature of PGPal. It compares the lifestyle preference attributes stored in each tenant's profile and produces a compatibility score (0–100+) for every pair.

9.1 Preference Attributes

  • Sleep Schedule: "Early Bird", "Night Owl", or "Flexible"
  • Guest Policy: "No Guests", "Occasional", or "Frequent"
  • Hobbies: Free-form list e.g. ["Reading", "Gaming", "Music"]

9.2 Scoring Weights

AttributePoints if MatchedRationale
Sleep Schedule50Most impactful lifestyle factor for cohabitation
Guest Policy30Directly affects shared space comfort
Each Shared Hobby10 eachBonus social compatibility points

9.3 Algorithm

Input: Preferences of the logged-in tenant (P1) and all other tenants (P2 ... Pn)

For each other tenant Pi:
  score = 0
  IF P1.sleepSchedule == Pi.sleepSchedule → score += 50
  IF P1.guestPolicy == Pi.guestPolicy → score += 30
  sharedHobbies = intersection(P1.hobbies, Pi.hobbies)
  score += |sharedHobbies| × 10

Output: All tenants sorted in descending order of score

9.4 Implementation

The algorithm runs server-side in backend/routes/matcher.js. This prevents exposing all tenant preferences to the client. The route is protected by JWT middleware, ensuring only authenticated tenants can request matches. Results are returned as a JSON array with each entry containing the tenant's name, room number, score, and list of shared hobbies.

11. Security Implementation

  • Password Hashing: All passwords are hashed using bcryptjs with a salt round of 10 before being stored. Plaintext passwords are never persisted.
  • JWT Authentication: On successful login, a JWT is issued signed with a secret key stored in environment variables (.env). The token expires in 24 hours.
  • Middleware Guards: Every API route (except login and register) passes through verifyToken middleware. Admin-only routes additionally pass through adminOnly middleware.
  • Role Separation: Even if a tenant guesses an admin route URL, the server returns HTTP 403. The JWT payload is decoded on every request — roles cannot be spoofed client-side.
  • Client-side Expiry Handling: An AngularJS HTTP response interceptor watches for 401 responses. When detected, it clears localStorage and redirects to the login page automatically.
  • Environment Variables: The MongoDB URI and JWT secret are stored in a .env file excluded from version control via .gitignore.
  • Password never returned: All Mongoose queries that return tenant data use .select('-password') to exclude the hashed password field from API responses.

12. Conclusion

PGPal successfully demonstrates a complete, functional web application built on the MEAN stack. The system covers all aspects of PG management from room allocation to rent verification, while the Roommate Matcher feature introduces algorithmic logic that goes beyond basic CRUD functionality.

The application follows industry-standard practices including JWT-based stateless authentication, RESTful API design, NoSQL schema design with Mongoose, and client-side single-page application architecture with AngularJS 1.x. The separation of admin and tenant modules through role-based access control reflects real-world software design principles.

The project demonstrates the practical application of all core MCA curriculum topics including database management, web application development, client-server architecture, and software security.

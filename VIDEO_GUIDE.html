Section 1 — Introduction

Start with a blank browser tab
Hello, my name is Abhiram. This is PGPal — a Paying Guest Management System built as my MCA project using the MEAN stack: MongoDB Atlas, Express.js, AngularJS version 1, and Node.js.
Type http://localhost:3020 in the browser
The application opens on the login page. PGPal has two roles — Admin and Tenant — each with their own dedicated set of features. All communication between the frontend and backend happens through RESTful APIs using JSON.

Section 2 — Admin Login and Dashboard

Step 1 — Login as Admin
Type admin@pgpal.com and Admin@123, click Login
I'm logging in as the admin. The backend validates the credentials, hashes the password using bcryptjs, and on success returns a JSON Web Token which the frontend stores in localStorage and attaches to every future API call automatically.
Step 2 — Admin Dashboard
Point to the stat cards one by one
The admin dashboard gives a complete overview at a glance. We have total rooms, occupied and vacant counts, the bed-based occupancy rate, expected monthly revenue — which is price per person multiplied by occupied beds — and the confirmed rent collected this month from tenant receipts. Below are the most recent maintenance requests and newly registered tenants.
Pause on the "Confirmed This Month" vs "Expected Revenue" cards — this is a good point to explain the business logic.

Section 3 — Room Management

Step 3 — Navigate to Rooms
Click "Rooms" in navbar
The Rooms page shows all PG rooms with their type, price per person, current occupancy, amenities, and status. I can search rooms using the search bar here.
Step 4 — Show Add Room
Fill the form on the right: Room "301", Type "Triple", Price "2500", add amenity "Fan"
Adding a new room — I select "Triple" as the type. Notice that capacity is automatically set to 3. This is handled by a Mongoose pre-save hook in the Room model that derives capacity from the room type. This ensures data consistency — the admin cannot accidentally set a wrong capacity.
Click Add Room, then click the 👥 button on an occupied room
Clicking this button shows the current occupants of that room — their name, email, phone, and move-in date.

Section 4 — Tenant Management

Step 5 — Tenants Page
Click "Tenants" in navbar
The Tenant Management page lists all registered tenants. I can see the room preferences they selected during registration — this tells me what type of sharing they are comfortable with. The "Next Rent Due" column shows the actual calendar date computed from their rent due day setting.
Step 6 — Assign Room
Select a vacant room from the dropdown and click Assign for a tenant
Assigning a room to a tenant. The dropdown only shows vacant rooms — rooms that still have available beds. When assigned, the room's occupied bed count increments and the status is automatically recalculated by the Mongoose hook. The tenant's move-in date is also updated to today.
This is a great point to explain the pre-save hook and how it prevents stale data.
Step 7 — Edit Tenant / Vacate
Click Edit on a tenant row, change the Rent Due Day, save
The row becomes editable inline. I can update name, phone, or rent due day without leaving the page. The Vacate button moves a tenant out of their room without deleting their account.

Section 5 — Maintenance and Notices

Step 8 — Maintenance Log
Click Maintenance in navbar
The admin sees all maintenance complaints filed by tenants. Each complaint has a status — Pending, In Progress, or Resolved. Clicking Update opens a form where the admin can change the status and add a note that the tenant will see on their dashboard.
Click Update on a request, change to "In Progress", add a note, save
Step 9 — Notices
Click Notices, post a notice with a title and message
The admin can post text-based notices. These appear on every tenant's Notice Board and are also previewed on the tenant dashboard. Tenants cannot post notices — only view them.

Section 6 — Rent Receipts (Admin side)

Step 10 — Review Receipts
Click Rent in navbar
This is the Rent Receipt management page. When a tenant makes a payment, they submit a claim here with the month, amount, and their UPI or bank reference number. The admin can then confirm or reject the payment with a note. This creates a transparent, trackable rent history without a full payment gateway.
Click Review on a Pending receipt, select Confirmed, save
When confirmed, this payment is counted in the "Confirmed This Month" figure on the dashboard.

Section 7 — Tenant Login and Dashboard

Step 11 — Logout and login as Tenant
Click Logout, login with arjun@pgpal.com / Arjun@123
Now logging in as a tenant. The AngularJS router automatically redirects to the tenant dashboard based on the role field inside the JWT payload.
Step 12 — Tenant Dashboard
Point to each stat card
The tenant dashboard shows their name and contact, the assigned room with price per person, a countdown of days until the next rent is due, their open complaint count, and the status of their last rent receipt. Below are the latest notices from admin and their complaint history.
Click the profile card to open the edit form
Tenants can edit their name, phone, and password inline. They can also update their roommate preferences — sleep schedule, guest policy, and hobbies — which feeds into the Roommate Matcher.
Step 13 — Submit Maintenance Request
Click Maintenance, fill in title and description, submit
A tenant can file a maintenance complaint from their maintenance page. It goes to the admin's log immediately and they can track the status here as the admin updates it.
Step 14 — Submit Rent Receipt
Click Rent, fill the form with current month, amount, and a reference number
The tenant submits their payment claim here. They select the month, enter the amount they paid, and their UPI transaction ID or bank reference. The system prevents submitting two claims for the same month. The admin then verifies and confirms or rejects it.

Section 8 — Roommate Matcher (Algorithm Highlight)

Step 15 — Roommate Matcher
Click Roommate Matcher, let the cards load
This is the Roommate Matcher — the special logic feature of PGPal. The algorithm runs on the server side. It compares the logged-in tenant's preferences against every other tenant using a weighted scoring model.
Point to your own preferences box at the top, then point to the match cards
Sleep schedule carries 50 points — it's the most critical lifestyle factor for cohabitation. Guest policy carries 30 points. Each shared hobby adds 10 bonus points. Results are sorted from most to least compatible. Green badge means high compatibility, yellow is moderate, red is low. The shared hobbies are explicitly listed on each card.
This is the most unique part of the project — spend 30 extra seconds explaining the weights. Mention that the algorithm is extensible — more attributes can be added by just adding weights.

Section 9 — Additional Features and Closing

Step 16 — Dark Mode
Click the ☾ Dark button in navbar
The application supports a carbon black dark mode. The preference is saved in localStorage so it persists across sessions and page refreshes.
Step 17 — Print Report (as Admin)
Login as admin, go to Tenants, click the Print button
Admins can print the full tenant list. Print-specific CSS hides the navigation, buttons, and forms and shows only a clean formatted table suitable for record-keeping.
Step 18 — Closing
PGPal demonstrates a complete, full-stack MEAN application covering authentication, role-based access control, CRUD operations, relational data with MongoDB, and a custom matching algorithm. Thank you.

Key Points to Mention if Asked in Viva

QuestionAnswer
Why AngularJS 1.x and not Angular 2+?Project specification required AngularJS 1.x with ng-app, $scope, and controllers. The architecture remains MVC on the client side.
Why MongoDB instead of SQL?PG data is document-oriented. Tenant preferences are stored as nested JSON objects — this maps naturally to MongoDB documents. Schema flexibility also allowed adding the RentPayment feature without migrations.
How does JWT work here?Login returns a signed token with user ID and role. Every API request sends it as Bearer header. The server verifies the signature — no session storage needed, making it stateless.
What is the Mongoose pre-save hook for?It automatically recalculates room capacity from type and updates room status (Vacant/Occupied) whenever a room document is saved. This eliminates manual status management.
How does role-based access work?Two middleware functions: verifyToken (checks JWT) and adminOnly (checks role field in decoded token). Admin routes use both. Tenant routes use only verifyToken.
What is the Roommate Matcher algorithm?Weighted scoring: sleep schedule (50pts), guest policy (30pts), shared hobbies (10pts each). Runs server-side. Results sorted descending by score.
What database is used and why cloud?MongoDB Atlas — cloud hosting means the app works from any machine without local DB setup. Ideal for academic demo environments.
